# AI-Morning-Brief

> 매일 아침, 쏟아지는 AI 업계 소식 중 **"나에게 중요한 것"만** 골라
> Discord로 한 장 브리핑을 보내주는 **무인 자동화 파이프라인**.
>
> Python CLI · SQLite · Claude API · launchd — **현재 매일 04:30 실제 운영 중.**

---

## 1. 왜 만들었는가 (Problem)

AI 도구로 제품을 만드는 1인 운영자에게 **트렌드를 놓치는 것은 곧 경쟁력 상실**이다.
그런데 매일 아침 Hugging Face·GitHub·Simon Willison·TechCrunch… 7~8개 사이트를 손으로 도는 일은 현실적으로 지속 불가능했다.

핵심 문제는 양이 아니라 **선별**이었다.
- 기사는 많지만, *나의 일(AI 자동화 공방 운영)에 바로 쓸 만한 것*은 극소수다.
- "중요도"는 사람마다 다르다. 업계 일반 중요도와 **내 관점의 관련도**는 별개다.
- 그리고 이 일은 **내가 자는 새벽에도 알아서** 돌아가야 의미가 있다.

> 한 줄: *"내가 매일 아침 수동으로 하던 정보 수집·선별을, 내 관점을 아는 무인 파이프라인에게 넘기고 싶었다."*

---

## 2. 어떻게 해결했는가 (Solution)

UI 없는 **헤드리스 자동화 프로세스**로 설계했다. 결과는 Discord 메시지와 로컬 Markdown 리포트로 받으면 되므로, 화면을 만드는 비용을 전부 제거하고 **가장 빠르게 검증 가능한 형태**를 골랐다.

```
RSS 수집 → SQLite 저장 → Claude 분석/선별 → Markdown 리포트 → Discord 전송
   │           │             │ (운영자 프로필을 '렌즈'로 주입)
   └ 부분 실패 무시  └ 중복 차단     └ 실패 시 키워드 요약으로 자동 강등
```

가장 중요한 설계 결정은 **`operator_profile.md`(운영자 프로필)를 Claude 분석의 필수 입력으로 주입**한 것이다.
"나는 Claude/MCP/AI Agent/자동화에 관심이 크고, 단순 주가·가십은 낮게 본다"는 프로필을 프롬프트의 렌즈로 넣어, Claude가 **업계 중요도(importance)와 내 관점 관련도(relevance)를 분리해 0~5점으로 평가**하게 했다.

그리고 무인 실행을 위해 macOS **launchd**로 매일 04:30에 자동 기동한다. (cron은 슬립 중 작업을 영구히 건너뛰지만, launchd는 깨어난 직후 놓친 작업을 실행 → catch-up 정책과 맞물린다.)

## 3. 핵심 기능

| 기능 | 설명 |
|------|------|
| **일간 브리핑** | 지정 RSS를 수집 → Claude가 기사별 요약·태그·중요도·관련도 평가 → 핵심 변화/적용 아이디어/액션을 한 장으로 |
| **주간 흐름 리포트 (`--weekly`)** | 7일치를 집계해 *지속 줄기 vs 단발 버스트*, 노이즈 소스, 공방 적용 픽을 ISO 주차 단위로 종합 |
| **무중단 fallback** | API 키·모델명·네트워크가 실패해도 키워드 기반 요약으로 자동 강등 → **파이프라인은 절대 멈추지 않음** |
| **소스 진단 (`--check-sources`)** | 실행 전 RSS 접근성을 읽기 전용으로 점검(HTTP 상태·기사 수) |
| **catch-up 수집** | Mac이 슬립이어서 놓친 날도 다음 실행이 그 시점부터 보충 수집 |
| **무상태 dry-run (`--dry-run`)** | DB·리포트·Discord 어떤 것도 건드리지 않고 실행 계획만 미리보기 |

## 4. 기술적으로 어려웠던 점

1. **"무인이면 절대 죽으면 안 된다"는 제약**
   새벽에 혼자 도는 프로그램은 누가 봐주지 않는다. 그래서 **"수집·저장은 분석(Claude) 성공 여부와 무관하게 항상 먼저 끝낸다"**는 불변식을 세웠다. Claude가 죽어도, Discord가 죽어도, 단일 소스가 타임아웃 나도 데이터 유실은 0이고 파이프라인은 끝까지 간다.

2. **macOS launchd × TCC 보호 폴더 함정 (실제 운영 중 발견)**
   개발은 됐는데 스케줄 실행만 매번 `Operation not permitted`로 실패했다. 원인은 macOS가 `~/Desktop`을 개인정보(TCC) 보호 폴더로 막기 때문. **운영 정본을 보호 폴더 밖(`~/AI-Morning-Brief-run/`)으로 분리**하고 rsync로 동기화하는 구조로 해결했다. — *문서가 아니라 실기동에서만 드러나는 문제였다.*

3. **주간 트리거를 "카운터"가 아니라 "ISO 주차 커서"로**
   "일간 7번 돌면 주간 생성"식 카운터는 슬립으로 하루 거르거나 빈 리포트가 끼면 정렬이 깨진다. 대신 **현재 날짜에서 ISO 주차를 역산하는 커서** 방식으로 바꿔, 일요일에 늦게 깨어나도 같은 주를 정확히 가리키고 중복 생성을 막는 **자가치유 트리거**를 만들었다.

4. **테스트가 운영 데이터를 절대 오염시키지 않게**
   `MORNINGBRIEF_DB/RAW_DIR/REPORTS_DIR` 환경변수 override를 처음부터 설계에 박아, pytest가 운영 DB·리포트와 **완전히 분리된 경로**에서만 돌도록 격리했다.

## 5. 내가 직접 설계한 부분

> 이 프로젝트의 코드 구현은 AI 에이전트가 했지만, **무엇을 만들지·어떤 구조로 갈지·어디서 멈출지의 결정은 내가 소유**했다.

- **형태 결정**: "UI를 만들 것인가"를 버리고 **헤드리스 자동화**로 확정 — 검증 비용을 최소화하는 형태 선택.
- **실패 복구 불변식**: "저장 먼저, 분석 나중, 무엇이 죽어도 데이터는 남는다"는 핵심 규칙을 직접 정의.
- **운영자 프로필을 분석의 렌즈로** 쓰는 구조 — 범용 요약기가 아니라 *내 관점의 비서*로 만든 결정.
- **운영 위치 분리(TCC 회피)**: 실기동 장애를 직접 진단하고 개발/운영 경로를 나누는 운영 구조를 설계.
- **주간 커서 트리거**: 카운터의 함정을 짚고 ISO 주차 커서라는 대안을 채택한 판단.
- **운영 중 버그를 직접 잡음**: 실제 전송에서 "한겨레 소스 210건·평균 중요도 0.9 → 최대 노이즈원" 같은 문제를 발견하고 수정 이슈로 연결.

## 6. AI를 어떻게 활용했는가

이 프로젝트는 **AI가 만든 산출물이자, AI 생산 공정(IT상상공방) 자체의 검증 사례**다.

- **다중 에이전트 파이프라인**으로 생산: 상담 → 기획 → 설계 → 이슈등록 → 개발 → 자동 QA → 납품을 단계별 전담 에이전트가 처리.
- **모델을 일에 맞게 배치**: 무거운 판단(설계)은 고성능 모델, 반복 구현은 경량 모델(Sonnet)로 — 비용을 일의 무게에 정렬.
- **FEAT 문서 1:1 매칭**: 작업을 "이슈 1개 = 설계 문서(FEAT) 1개"로 쪼개, 개발 에이전트가 **딱 그 문서 하나만 읽고** 구현하게 해 토큰·탐색 비용을 줄임.
- **Claude를 런타임에도 활용**: 수집된 기사를 Claude API로 분석·선별(JSON 구조화 출력)하되, 실패하면 규칙 기반 fallback으로 강등되는 **이중화** 구조.
- **AI가 못 잡는 건 사람 게이트로**: 방향 결정·실사용 검증은 내가 직접 승인. AI는 실행, 사람은 결정.

## 7. 프로젝트를 통해 배운 점

- **무인 시스템의 품질은 "성공 경로"가 아니라 "실패 경로 설계"에서 갈린다.** 잘 될 때가 아니라 죽을 때 어떻게 버티는지가 전부였다.
- **문서로는 안 보이고 실기동에서만 보이는 문제가 있다** (TCC 보호 폴더). "동작 검증"은 반드시 실제 운영 환경에서 끝까지 가봐야 한다.
- **AI에게 좋은 결과를 받으려면, AI가 읽을 명세(FEAT)의 품질이 곧 결과 품질**이다. 잘 쪼갠 작업 단위와 명확한 입력이 토큰·재작업을 동시에 줄인다.
- **자동화의 가치는 "한 번 됨"이 아니라 "매일 알아서 됨"**에 있다. 실제로 매일 아침 브리핑이 도착하기 시작하면서야 도구가 완성됐다고 느꼈다.

## 실행 방법 (요약)

```bash
cd projects/AI-Morning-Brief/05-개발
pip install -r requirements.txt

# 설정
cp config/.env.example .env                       # ANTHROPIC_API_KEY, DISCORD_WEBHOOK_URL 입력
cp config/sources.example.yaml config/sources.yaml # 구독할 RSS 목록

# 외부 의존 없이 파이프라인 점검
python main.py --test --no-discord

# 실제 1회 실행 (수집 → 분석 → Discord 전송)
python main.py

# 주간 흐름 리포트
python main.py --weekly
```

- 키가 없어도 **fallback 요약으로 정상 동작**한다(파이프라인 중단 없음).
- macOS 무인 스케줄(launchd) 등록, TCC 보호 폴더 회피, 운영 배포본 동기화, 문제 해결 등 **상세 운영 매뉴얼은 [`RUNBOOK.md`](RUNBOOK.md)** 참조.
